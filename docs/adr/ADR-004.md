# ADR-004 — Dual WASM builds, feature-detected

**Status:** accepted  
**Date:** 2026-07-02

## Context

SIMD128 is not universally available: Safari < 16.4 and some older Node.js versions lack it. Shipping only a SIMD build excludes those environments; shipping only a scalar build sacrifices performance everywhere else.

## Decision

Build **two WASM binaries** for every kernel set:

1. `*.simd.wasm` — compiled with SIMD128 enabled (`-msimd128` / `+simd128`).
2. `*.scalar.wasm` — compiled without SIMD, serving as the fallback.

At **load time**, probe SIMD availability using `WebAssembly.validate` on a minimal hand-written SIMD module. Based on the result, dynamically import (`import()`) the appropriate binary. The detection and import logic lives in `src/memory/` and runs once per page/process load.

## Implications

- CI must build and test **both** variants on every commit.
- The size gate (≤ 75 KB gzipped per binary) applies to each variant independently.
- The per-kernel bench gate (§5: SIMD ≥ 1.5× typed-array baseline) only applies to the SIMD variant; scalar is expected to be near parity with the typed-array baseline.
- All kernel authors must write both scalar and SIMD implementations. The wasm build scripts handle the two compilation targets.

## Alternatives considered

- **SIMD-only:** Simpler build but excludes Safari < 16.4 (non-trivial market share as of 2026).
- **Runtime polyfill / JS scalar fallback:** Avoids a second binary but at a large performance cost; also increases JS bundle size.
- **Wasm feature detection via try/catch instantiation:** More heavyweight than `WebAssembly.validate`; validate is the standard idiom for this use case.
