# ADR-005 — No index; hash-based relational ops

**Status:** accepted  
**Date:** 2026-07-02

## Context

Pandas' index/alignment semantics are a frequent source of confusion and bugs. Relational operations (groupby, join) require either sort-based or hash-based strategies.

## Decision

**No row index.** Row position is identity. There are no index alignment semantics, no automatic broadcasting by label, and no `.loc` / `.iloc` distinction.

**Relational ops use WASM hash tables:**

- `groupby` builds a 64-bit hash table over the group-key column values, then dispatches aggregation kernels (from Phase 2 Agent B) per group.
- `join` (inner / left) uses the same hash-table approach keyed on the join column(s).
- Strings hash their **dictionary indices** (after dictionary unification), not the raw UTF-8 bytes, so hashing is O(1) regardless of string length.
- `sortValues` produces an `i32` permutation via `argsort` (stable, multi-key), then applies `gather` to materialize the sorted frame.

## Implications

- Dictionary unification is a prerequisite for string joins/groupby across two frames. The unification step must be implemented in Phase 1 (`src/memory/`).
- The `argsort` + `gather` pattern means a sorted view is always a new materialized frame, not a lazy view. This is consistent with the "all ops return new frames" principle (see §1 non-goals).
- `count` counts non-null rows; `size` counts all rows (ADR-002 validity bitmap makes this distinction free).

## Alternatives considered

- **Sort-merge join:** Lower memory overhead for large sorted inputs, but requires materialized sort passes. Hash join is simpler to implement in WASM and faster for unsorted data.
- **Pandas-style index:** Aligns rows by label, enabling implicit joins. Too much implicit behavior; we prefer explicit `join()` calls.
