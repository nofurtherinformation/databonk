# ADR-009 — i64 / BigInt columns

**Status:** accepted
**Date:** 2026-07-03
**Deciding agent:** v2 LEAD (bead dataframe-dh9.1), commissioned by Dylan for 0.2.0.
**Reverses:** the v1 non-goal "i64 / BigInt columns" (spec §0 non-goals; `dtypes.md`
§1 "Non-goals (v2)").

---

## Context

v1 shipped `f64 f32 i32 u32 bool utf8` and explicitly deferred 64-bit integers
(spec §0: "i64 / BigInt columns … (v2)"). The v1 parking lot (`status.md`) recorded
the blockers: a wasm32 ABI extension for 64-bit values crossing the boundary, and
unknown JS `BigInt` boundary overhead.

We reverse that non-goal now because:

1. **It is the prerequisite for temporals (ADR-010).** `timestamp` is stored as an
   `i64` count of epoch milliseconds; without an i64 physical type there is no
   timestamp type. i64 must land first (dependency `dh9.3 → dh9.4 → dh9.5`).
2. **64-bit integer columns are a first-class analytics need**: database bigint IDs,
   large counts/sums that overflow `i32`, and epoch values — all common in the CSV /
   Arrow / Parquet sources v2 targets.
3. The boundary-overhead unknown is now answerable: `BigInt64Array` gives a zero-copy
   typed-array over wasm memory (same ADR-001 `viewOf` model as every other dtype),
   so the only `BigInt` allocation cost is at the *scalar* boundary (literals,
   reduction results, per-row escape-hatch access), not the bulk columnar path.

Reversing a locked non-goal requires this ADR (spec §2).

## Decision

Add a sixth numeric dtype, `i64`: a signed 64-bit integer column.

### Storage & boundary

- **Physical storage:** contiguous `i64`, 8 bytes/value, 16-byte-aligned (ABI §3),
  in wasm linear memory. Validity is the standard Arrow LSB bitmap (ABI §4.1) — null
  is **never** a sentinel value, identical to every other dtype.
- **JS TypedArray:** `BigInt64Array` over `memory.buffer` (the `viewOf` layer,
  ADR-001). This is the **only** new `TypedArray` kind v2 introduces.
- **Boundary type = `bigint`.** `columnToArray`/`toArray` yields `bigint | null`.
  Every scalar that crosses the wasm boundary as a wasm `i64` — kernel scalar
  operands, `fill` values, and i64 reduction *returns* — surfaces in JS as a
  `bigint` (wasm `i64` params/returns marshal to JS `BigInt`; see ABI §5/§9 v2).
- **Column-construction literals** accept `number | bigint`:
  - a `bigint` is used directly (wrapped to 64 bits via two's complement if out of
    `[-2^63, 2^63-1]` — same wrapping model as arithmetic below);
  - a `number` must be an exact integer in the safe-integer range
    (`Number.isSafeInteger`, i.e. `|x| ≤ 2^53-1`). A non-integer or out-of-safe-range
    `number` **throws** a helpful error naming the offending value (silent precision
    loss is worse than a throw). `null`/`undefined` are nulls; a `NaN` `number` is
    **not** a null here — it is an invalid i64 literal and throws.

### Arithmetic — wrapping two's-complement (mod 2^64)

- `add_i64 sub_i64 mul_i64 neg_i64` **wrap** on overflow (two's complement, mod 2^64),
  matching Rust `wrapping_add/…` and JS `BigInt.asIntN(64, x)`. No trap, no null on
  overflow.
- `sum_i64` is likewise **wrapping** and therefore order-insensitive (wrapping i64 add
  is associative and commutative mod 2^64); it returns an `i64` (→ `bigint` in JS),
  matching the "integer sum stays integer" precedent (`dtypes.md` §3.2, ABI §9 reduction
  note).
- `div_i64 mod_i64` are **truncating** integer division/remainder (toward zero). A
  **zero divisor → null** (validity `0`), never a trap — identical to `i32`/`u32`
  (`dtypes.md` §3.2, ABI §7). `mod` follows Rust remainder sign (result takes the sign
  of the dividend). Overflow of `i64::MIN / -1` wraps to `i64::MIN` (guarded, no trap).
- `mean_i64 std_i64 var_i64` return **`f64`**, not `i64`: each value is converted
  `i64 → f64` first (rounding beyond ±2^53, see the cast rule) and then accumulated
  with the *float* reduction strategy. **Precision note (documented, load-bearing):**
  i64 magnitudes above 2^53 lose precision in `mean/std/var` because the f64 mantissa
  is 53 bits; this is the same trade-off v1 already makes for `mean_i32`/`mean_u32`
  (which also return `f64`), extended to a wider range.

### Mixed-dtype arithmetic (implicit widening)

v1's single implicit rule (int→float widening) broadens to a **widening lattice**.
The complete set of *newly allowed* mixed pairs:

| mixed pair | result | conversion |
|---|---|---|
| `i64 ⊕ i64` | `i64` | none |
| `i32` or `u32` ⊕ `i64` | `i64` | int widened to `i64` (**exact**: i32 sign-extend, u32 zero-extend) |
| `i64 ⊕ f64` | `f64` | i64 widened to f64 (⚠ rounds if \|x\|>2^53) |
| `i64 ⊕ f32` | **`f64`** | **both** widened to f64 (i64→f64 rounds >2^53; f32→f64 exact) — result is f64, **not** f32 |

`i64 ⊕ f32 → f64` is a deliberate asymmetry vs. v1's `i32 ⊕ f32 → f32`: an f32's
24-bit mantissa cannot meaningfully hold an i64, so the result widens to the larger
float. Every **other** mixed pair with `i64` (`i64 ⊕ bool`, `i64 ⊕ utf8`) stays an
**error** requiring an explicit cast, exactly like v1's unsupported-mixed rule.
`i32 ⊕ u32` and `f32 ⊕ f64` remain errors (unchanged).

### Explicit casts

Full cell semantics are in `dtypes.md` §2 (v2). The load-bearing rules:

- **`f64 → i64` / `f32 → i64`:** truncate toward zero; **out-of-range** (`|x| ≥ 2^63`)
  or `NaN` → **null** (⚠), never trap — same failure model as `f64 → i32`.
- **`i64 → f64` / `i64 → f32`:** exact for `|x| ≤ 2^53`; **beyond ±2^53 it ROUNDS to
  the nearest representable float and is NOT nulled** (matches JS `Number(bigint)`).
  This is deliberately different from the ⚠ float→int casts: widening never fails, it
  only loses precision. Documented so no agent adds a spurious range-null.
- **`i64 → i32` / `i64 → u32`:** **wrap-truncate** to the low 32 bits (two's
  complement), never null, never trap. `i32 → i64` / `u32 → i64` are exact widenings.
  (This differs from v1's `i32 → u32` range-null cast; i64↔narrow-int follows i64's
  wrapping-arithmetic model. See the `dtypes.md` §2 note.)
- **`bool → i64`** (`t→1n, f→0n`, exact) and **`i64 → bool`** (`x≠0 → true`,
  `0 → false`), mirroring the v1 bool↔i32 rows. **`i64 ↔ utf8`** is ✗ (throws), like
  every numeric↔utf8 pair.

### SIMD

Per ADR-004 both builds export identical i64 symbols; the SIMD build uses `i64x2`
where the ISA provides it:

- **`add/sub/neg` and all six comparisons** (`i64x2.add/sub/neg`, `i64x2.eq/ne/
  lt_s/gt_s/le_s/ge_s`): 2 lanes/iteration + scalar tail.
- **`mul/div/mod` are scalar-only in BOTH builds.** SIMD128 has **no** vector integer
  division/remainder at all, and `i64x2.mul` offers no throughput benefit (it is
  emulated as scalarized 32-bit partial products on mainstream engines). Following the
  existing `hash.rs` precedent, these three stay scalar. The scalar build already runs
  them scalar, so both binaries agree bit-for-bit.

### Hashing (relational ops, ADR-005)

`hash_i64` mixes the 64-bit value directly through the **splitmix64** finalizer to a
64-bit hash. Null rows hash to the reserved `H_NULL` constant (ABI §9 Agent D), so
nulls form one group and never join-match — unchanged null semantics. `hash_combine`,
`group_build`, and both `join_*` kernels operate on 64-bit hash values and are
**dtype-agnostic**: they are reused unchanged (no i64 variants).

### The rest of the kernel families

`filter_i64 gather_i64 argsort_i64 topk_i64 first_i64 last_i64 min_i64 max_i64
count nunique_i64` follow the existing per-dtype patterns (ABI §9 v2). `argsort_i64`
takes the v1.2 caller-provided `scratch_ptr` (`i32[len]`). `count` is validity-only
(`count_null`) and needs no i64 variant. `nunique_i64` **may retain the documented
O(n²)-beyond-scratch ceiling** already accepted for other dtypes (`status.md` parking
lot) — it is not required to improve on it.

Total order for `argsort`/`topk`/`min`/`max` is signed-i64 numeric order with **nulls
last** in both directions (`dtypes.md` §4.6, unchanged — i64 has no NaN concern).

## Consequences

- One new `TypedArray` kind (`BigInt64Array`) enters `memory.d.ts`
  (`ViewDType`/`ColumnView`/`TypedArrayCtor`), `Cell` gains `bigint`, and `DType`
  gains `'i64'`. Everything else in the memory layer is width-parametric already.
- The `viewOf` bulk path stays zero-copy; the **only** `BigInt` cost is at scalar
  crossings (literals, reduction returns, row-proxy access in the escape hatch). The
  escape hatch's per-row `bigint` boxing is documented as slow, consistent with
  ADR-003's "escape hatch is the slow path."
- Binary-size impact is bounded: i64 elementwise/reduction/selection kernels are
  width variants of existing code; the 75 KB-gzipped per-binary budget (ABI §1) is not
  at risk (v1 binaries sit ~84–132× under budget, ADR-007). CI size gate still guards.
- Conformance fixtures (dh9.2) **must** cover: wrapping overflow (add/mul/sum), the
  `i64→f64` >2^53 rounding, `f64→i64` out-of-range→null, `i64→i32/u32` wrap, div/mod
  zero→null, and the `number` safe-int literal throw. The SIMD null-lane trap called
  out in ADR-007 applies to the i64 reductions too.

## Alternatives rejected

- **Sentinel/`BigInt`-only storage (no `BigInt64Array`, box every value):** kills the
  zero-copy columnar model (ADR-001) — O(n) `BigInt` allocations per column crossing,
  failing the perf targets. Rejected.
- **Emulate i64 as two `i32` lanes (lo/hi) to avoid the wasm-i64 boundary:** wasm32
  supports `i64` value types natively; splitting them doubles kernel complexity and
  loses `i64x2` SIMD. No upside. Rejected.
- **`number`-only literals (reject `bigint` inputs):** cannot express values above
  2^53, defeating the point of i64. Rejected — we accept both and throw on unsafe
  `number`.
- **Saturating or trapping i64 arithmetic:** inconsistent with v1 `i32`/`u32` (which
  wrap) and with JS `BigInt.asIntN`. Wrapping is the least-surprising, trap-free
  choice. Rejected.
- **`mean/std/var_i64 → i64` (fixed-point):** meaningless for a mean; every consumer
  expects a real number. `f64` with a documented precision note matches v1's integer
  reductions. Rejected.
