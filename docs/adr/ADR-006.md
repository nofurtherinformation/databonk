# ADR-006 — Parallelism is an opt-in shared-memory mode

**Status:** accepted  
**Date:** 2026-07-02

## Context

WebAssembly linear memory cannot be transferred to a Worker via `postMessage` (unlike `ArrayBuffer`, a `WebAssembly.Memory` is not transferable). True data parallelism requires a `SharedArrayBuffer`-backed `WebAssembly.Memory`, which in turn requires cross-origin isolation (COOP/COEP headers).

Cross-origin isolation is not universally available (some hosting environments cannot set the required headers), so mandatory parallelism would silently break or degrade in those environments.

## Decision

**v1 main path is single-threaded.** No threads, no SharedArrayBuffer by default.

**Opt-in parallel mode** (`enableThreads()`):

- Switches to a shared `WebAssembly.Memory` (SAB-backed) build variant.
- Spawns a worker pool.
- Dispatches chunked work for elementwise and reduction kernels across workers.
- At call time, detects whether `crossOriginIsolated === true` (browser) or SAB is available (Node). If isolation is absent, `enableThreads()` emits a console warning and no-ops — the library continues running single-threaded.

**Ship in Phase 5.** All code before Phase 5 must not assume threads or SAB.

## Implications

- The shared-memory build is a separate WASM compilation target (threads enabled), not a runtime switch of the existing binary.
- The worker pool and dispatch logic lives in `src/workers/` (currently stub).
- Phase 5 gate: correctness parity with single-thread; ≥ 1.8× speedup on 4 workers for 10M-row reductions; zero behavior change when the flag is off.
- All Phases 1–4 code must be thread-unaware. Writing to shared state without locks would be a bug; auditors should flag any global mutable state introduced before Phase 5.

## Alternatives considered

- **Always-parallel (SharedArrayBuffer mandatory):** Breaks in environments without COOP/COEP. Non-starter for a general-purpose npm library.
- **Worker-per-column (copy data):** Avoids SAB but copies the entire column buffer per worker dispatch — O(n) overhead per operation, negating the speedup.
- **Wasm threads (pthread emulation):** Emscripten pthread mode; heavy runtime, larger binary, complex debugging. The explicit worker-pool approach gives more control over chunk size and scheduling.
