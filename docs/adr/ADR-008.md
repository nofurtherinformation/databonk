# ADR-008 — Stable kernel ABI

**Status:** accepted  
**Date:** 2026-07-02

## Context

Phase 2 dispatches 3–4 agents in parallel to write different kernel families. Without a stable, agreed ABI between the JS API layer and the WASM kernel layer, agents would block on each other and integration would require extensive rewriting.

## Decision

All kernels are exported as **flat, C-style functions** operating on pointers and lengths. No objects, no closures, no indirect calls through vtables. Example signature:

```
add_f64(lhs_ptr: i32, rhs_ptr: i32, out_ptr: i32, validity_ptr: i32, len: i32) → void
```

Where:
- `lhs_ptr`, `rhs_ptr` — pointers into WASM linear memory for input data buffers (f64 arrays)
- `out_ptr` — pointer into WASM linear memory for the output buffer (pre-allocated by caller)
- `validity_ptr` — pointer to the validity bitmap (Arrow bit-packing convention, ADR-002); a null pointer (0) means "no nulls, skip bitmap check"
- `len` — number of elements

The complete ABI is documented in `/contracts/wasm-abi.md` (orchestrator-owned). `/contracts/wasm-abi.md` is the **contract** between the API layer (`src/kernels/`) and the kernel layer (`wasm/`), and is what allows kernel subagents to work independently without coordination.

## Implications

- Kernel agents (Phase 2) must not invent their own calling conventions. If a kernel requires a signature not in `wasm-abi.md`, the agent must STOP and escalate to the orchestrator to update the contract.
- The JS dispatch stubs in `src/kernels/` call these exports by name. Export names are stable identifiers; renaming requires a contract revision.
- The ABI document is read-only to all subagents. Only the orchestrator may update it, and only before dispatching the relevant Phase 2 agent.
- SIMD and scalar variants of the same kernel share the same ABI; the dynamic import (ADR-004) selects which binary is loaded, not which function is called.

## Alternatives considered

- **Object-oriented WASM interface (classes, vtables):** Easier to extend but incompatible with the flat export requirement of the WASM binary format and harder for agents to implement consistently.
- **Memory-mapped struct protocol:** Pass a pointer to a descriptor struct containing all parameters. More flexible for future extensions but harder to validate statically and requires layout agreement.
- **WASI / Component Model:** Standard but heavyweight for the binary size budget; also not yet universally supported in browsers.
