# ADR-003 — Hybrid query API

**Status:** accepted  
**Date:** 2026-07-02

## Context

Users need both ergonomic, high-performance operations and an escape hatch for arbitrary transformations not covered by the expression library.

## Decision

**Expression-first (fast path):** `col('a').gt(5)` builds a small AST that the compiler translates into a sequence of WASM kernel calls. This path is SIMD-vectorized and operates entirely in WASM linear memory, never touching JS per-row.

**Lambda escape hatch (slow path):** `df.filterFn(r => ...)` and `df.mapFn(...)` iterate rows via a reusable *row-proxy* object over column views (via `viewOf()`). No data is copied — but execution is at scalar JS speed because the JS engine cannot vectorize across WASM memory arbitrarily.

## Implications

- Documentation must clearly label the escape hatch as the slow path on every API page that exposes it.
- Every escape-hatch example in the docs must show the expression equivalent directly below.
- The row-proxy must be a single reused object (not a new object per row) to avoid GC pressure.
- String access through the row-proxy uses the JS-side memoized decode (per ADR-002), so it is slower than numeric access — this too must be documented.

## Alternatives considered

- **Expression-only (no lambda hatch):** Strictly better performance guarantees but significantly reduces ergonomics for uncommon transformations; likely to drive users to write their own loops anyway.
- **SQL string DSL:** Higher abstraction but requires a parser, adds bundle size, and is harder for AI agents to generate correctly.
