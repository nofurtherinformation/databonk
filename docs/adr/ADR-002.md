# ADR-002 — Arrow-compatible columnar layout

**Status:** accepted  
**Date:** 2026-07-02

## Context

Choosing a physical layout for columns in WASM memory determines: null-handling cost, interoperability with downstream tools (Apache Arrow IPC), and whether future I/O phases can be "nearly free."

## Decision

Per-column layout follows Apache Arrow conventions:

- **Data buffer:** contiguous typed array (e.g., `f64[]`, `i32[]`) in WASM linear memory.
- **Validity bitmap:** 1 bit per value, LSB ordering (Arrow convention). A zero bit indicates null.
- **Strings:** dictionary-encoded. An `i32` index buffer maps row positions to dictionary slots; the dictionary stores variable-length UTF-8 bytes via an `(offsets: i32[], bytes: u8[])` pair — all in WASM memory.
- **Decode memoization:** JS-side decoded strings are memoized per dictionary slot, so each unique string crosses the WASM boundary at most once per dictionary lifetime.

## Implications

- Layout compatibility is what makes Phase 6 Arrow IPC interop "nearly free" — Arrow IPC buffers can be memory-mapped directly into wasm memory with no reformatting.
- Dictionary unification (for `groupby` and `join` across frames) is a first-class operation because strings are encoded by slot, not by value.
- Validity bitmaps must be allocated alongside data buffers; null-aware kernels must test the bitmap before operating on a value.

## Alternatives considered

- **Sentinel nulls (NaN for f64, INT_MIN for i32):** Simpler but loses the ability to represent null in integer columns without widening the type, and conflates NaN-the-value with null.
- **Struct-of-arrays without bitmaps:** Forces null into a second parallel array of booleans, doubling the null storage cost vs. packed bitmaps and deviating from Arrow.
- **Row-oriented (array of objects):** Directly contradicts the "columnar" design goal; incompatible with SIMD kernels.
