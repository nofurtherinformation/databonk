# ADR-001 — WASM-core memory ownership

**Status:** accepted  
**Date:** 2026-07-02

## Context

Column buffers need to be accessible from both WASM kernels (for SIMD computation) and JavaScript (for API consumers). A naive approach would copy data across the JS↔WASM boundary on every operation, introducing unacceptable overhead at 1M+ rows.

## Decision

All column buffers are allocated inside the wasm module's linear memory via an exported allocator. JavaScript never owns column data; it holds `TypedArray` views constructed over `memory.buffer`. This makes the JS↔WASM boundary zero-copy in both directions.

**Consequence:** `memory.grow` detaches all existing `TypedArray` views (the underlying `ArrayBuffer` is replaced).

**Mitigation:** A module-level *generation counter* is incremented on every `memory.grow`. Every JS view accessor checks the counter and lazily rebuilds views when the counter has advanced. All view access goes through one `viewOf(column)` helper — no raw view caching anywhere else.

## Implications

- The `viewOf(column)` function is the single source of truth for constructing views. Direct caching of `TypedArray` objects outside this helper is forbidden.
- Tests must force `memory.grow` mid-operation and verify that views are correctly invalidated and rebuilt (see Phase 1 gate).
- Allocator exports: `alloc(size) → ptr`, `free(ptr)`, `realloc(ptr, newSize) → ptr`.

## Alternatives considered

- **JS owns buffers (SharedArrayBuffer passed into wasm):** Eliminates the grow-detach problem but requires SAB for every column, complicating cross-origin isolation requirements and adding overhead.
- **Copy-on-access:** Always copy data when crossing the boundary. Simple but violates the ≤ 25 KB JS size budget indirectly by requiring heavier bridging code, and fails performance targets.
