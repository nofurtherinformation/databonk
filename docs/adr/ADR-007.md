# ADR-007 — Implementation language for the WASM kernel layer

**Status:** accepted
**Decision:** **Rust** (`wasm32-unknown-unknown`, `no_std`, `panic = "abort"`), with the
dual SIMD128 / scalar build strategy of ADR-004 and the flat ABI of ADR-008.
**Date:** 2026-07-02
**Deciding agent:** Phase-0 LEAD (bead dataframe-99s.5), consuming the spikes from
dataframe-99s.3 (AssemblyScript) and dataframe-99s.4 (Rust).
**Supersedes:** the "pending" placeholder of the original ADR-007 transcription.

---

## Context

The kernel layer (ADR-008) is a set of flat, C-style WASM exports operating on
pointers + lengths. The library will accumulate *many* such kernels (elementwise,
reductions, selection/sort, relational hashing) authored largely by AI subagents in
Phase 2+. The choice of source language affects, in the spec's locked priority order:

1. **Kernel throughput** (the §5 gate: SIMD kernel ≥ 1.5× the typed-array JS baseline
   at 1M rows).
2. **Binary size** vs the ≤ 75 KB-gzipped-per-binary budget (§1, §5 size gate).
3. **Maintainability** for agents writing many kernels.

Per ADR-007's mandate, a spike implemented the same three kernels in **both**
AssemblyScript and Rust, each in **SIMD** and **scalar** variants, and measured them.

### The three spike kernels (identical ABI in both languages)

- `add_f64(a, b, out, len)` — elementwise f64 add (no validity; pure bandwidth test).
- `sum_f64_null(data, validity, len) -> f64` — null-aware reduction over an Arrow
  LSB validity bitmap (1 = valid).
- `cmp_gt_f64_mask(data, scalar, out_mask, len)` — compare-to-scalar producing an
  Arrow LSB bitmask (the hot first half of `filter`).

## Method & reproducibility

Both spikes were **rebuilt and re-benchmarked fresh, back-to-back, on the same machine**
by the LEAD (not merely read from the committed `results.json`). Environment: Docker
image `dataframe-dev`, Node v22.23.1, Rust 1.96.1 (`wasm32-unknown-unknown`),
`wasm-opt` v108, AssemblyScript via the spike's pinned `asc`. Both builds run
`wasm-opt -O3` as a post-pass (with `--enable-simd` on the SIMD build).

Bench protocol: warmup runs then N timed passes over buffers that live **inside** wasm
linear memory (the production zero-copy layout, ADR-001); `ms_mean` is the mean pass
time. Reported `Mops/s = rows / ms_mean` (million **elements** per second). Data is
all-valid for `sum` (validity bitmap = `0xFF`), deterministic pseudo-random f64 for
`add`/`cmp` with a scalar chosen for ~50 % selectivity.

### Correctness was verified before trusting any throughput number

A "fast wrong kernel is disqualifying." The AS spike ships a self-checking harness
(16-element fixture, validity `0x55`, mask fixture) that **passed** for both builds.
The Rust bench did **not** assert correctness, so the LEAD wrote
`wasm/spike/rust/verify.mjs` and ran it against both Rust builds.

**This surfaced a real bug:** the original Rust **SIMD** `sum_f64_null` returned
`128` instead of `64` on the null fixture. Root cause: for a pair where only one
element is valid, the kernel did `f64x2_splat(value)` — placing the value in *both*
lanes — then horizontally summed the two lanes at the end, **double-counting** every
singly-valid value. The benchmark uses all-valid data, so it always took the correct
`bits == 0x3` SIMD fast path and never exercised the buggy fallback; the throughput
number (3.0 Gops/s) was therefore measured on *correct* code, but the null path was
wrong. The LEAD fixed it (accumulate single values in a separate scalar accumulator
instead of splatting), rebuilt, re-verified (now `64`, PASS on both builds), and
re-benched — **fast-path throughput was unchanged** and the SIMD binary shrank
919 → 910 B. AS's SIMD `sum` used the correct `v128.and` mask pattern and was already
correct. Lesson recorded for Phase 2: the orchestrator's conformance fixtures **must**
include null-in-pair cases; both languages are equally capable of this bug.

All numbers below are the **post-fix, correctness-verified** fresh runs.

## Results

### Throughput at 1M rows (primary decision size)

`ms` = mean pass time (lower is better). `Mops/s` = million elements/second.

| Kernel            | AS scalar | AS SIMD | Rust scalar | Rust SIMD |
|-------------------|----------:|--------:|------------:|----------:|
| add_f64  (ms)     | 0.347     | 0.219   | 0.29        | **0.21**  |
| add_f64  (Mops/s) | 2 881     | 4 573   | 3 415       | **4 750** |
| sum_f64_null (ms) | **0.228** | 0.270   | 0.53        | 0.33      |
| sum_f64_null (Mops/s) | **4 392** | 3 699 | 1 901     | 3 000     |
| cmp_gt_mask (ms)  | **0.171** | 0.214   | 3.13        | 0.23      |
| cmp_gt_mask (Mops/s) | **5 860** | 4 670 | 320        | 4 253     |

### Throughput at 10M rows (confirms 1M is not a cache artifact)

| Kernel            | AS scalar | AS SIMD | Rust scalar | Rust SIMD |
|-------------------|----------:|--------:|------------:|----------:|
| add_f64 (Mops/s)  | 2 867     | 4 607   | 3 421       | 4 666     |
| sum_f64_null (Mops/s) | 4 329 | 3 841   | 1 941       | 3 025     |
| cmp_gt_mask (Mops/s)  | 6 024 | 4 722   | 316         | 4 268     |

### SIMD vs the typed-array JS baseline @ 1M (the §5 gate)

Baseline `bench/baselines/typedarray.json` @ 1M (ms/pass): `add_ab` = 2.069,
`filter_a_gt_0_5` = 3.685, `sum_a` = 0.869. Speedup = baseline_ms ÷ kernel_ms.

| Kernel (baseline op)          | Rust SIMD | AS SIMD | Gate (≥1.5×) |
|-------------------------------|----------:|--------:|:------------:|
| add_f64 (add_ab)              | 9.9×      | 9.4×    | PASS (both)  |
| sum_f64_null (sum_a)          | 2.6×      | 3.2×    | PASS (both)  |
| cmp_gt_mask (filter_a_gt_0_5) | 16.0×     | 17.2×   | PASS (both)  |

**Both languages clear the 1.5× per-kernel gate on every kernel by 2.6×–17×.**

Honesty caveats on the comparison (documented so later phases don't over-read it):

- **`add`**: the spike kernel does *not* handle validity, while JS `opAdd` does
  (`aValid & bValid` + writes `outValid`). The spike add does slightly less work; both
  are memory-bandwidth bound, so the ~9× reflects the JS-loop overhead ceiling, not a
  compute win. The production `add_f64` will still write an output validity bitmap.
- **`sum`**: the most apples-to-apples comparison (both null-aware). 2.6–3.2× is the
  most trustworthy single data point, and it clears the gate.
- **`cmp` vs `filter`**: the kernel emits *only* a bitmask; the JS `filter` baseline
  also **compacts** to an index array. So the 16–17× overstates the win for the
  *whole* filter — the gather/compaction is a separate Phase-2 (Agent C) kernel. Even
  so, comparison-to-bitmask is dramatically faster than a JS filter loop.

### Toolchain friction (from both `notes.md`)

| Dimension | AssemblyScript | Rust |
|---|---|---|
| SIMD source model | **Separate source files** required (no preprocessor; v128 intrinsics are syntax errors without `--enable simd`). Spike keeps `scalar.ts` + `simd.ts`. | **Single source**; `#[cfg(target_feature="simd128")]` gates SIMD in one file per kernel. |
| Auto-vectorization | Relies on `wasm-opt`; clean scalar code got autovectorized (scalar `cmp`/`sum` beat hand-written SIMD). | Same `wasm-opt` benefit; scalar `cmp` underperformed only due to a naive per-element memory RMW (fixable). |
| Standard library | **None** — hash tables, sort, all data structures hand-rolled. Big cost for Phase-2 Agents C (argsort/top-k) & D (hashing/groupby/join). | `core`/`alloc` + crates; native hash/sort algorithms available. |
| Native unit testing | No native path; must test in wasm. | `#[cfg(test)]` runs kernel logic natively before wasm. |
| Bit-manipulation ergonomics | Frequent explicit-cast errors (shift amounts must be `u32`/`u8`); friction on every bitmap kernel. | Integer widths explicit by construction; fewer surprises. |
| Output size | Smallest evaluated. | Small with `no_std` + `panic=abort`; can bloat if `std`/formatting leaks in. |
| Debugging | `--textFile` WAT readable; no native debugger; `trace()` for printf. | `wasm2wat`/`wasm-objdump`; native `#[test]` + `dbg!` off-wasm. |
| LLM authoring corpus | Smaller (AS is niche). | Large (systems-Rust is well represented) → fewer agent errors. |

## Decision — applying the locked priority order

**Criterion 1 — Kernel throughput (highest priority): effectively a tie; not decisive.**
Both languages clear the §5 gate on all three kernels (2.6×–17×). Head-to-head at the
SIMD level (the primary path under ADR-004) they are within measurement noise / hand-
tuning: `add` is a dead heat (0.21 vs 0.219 ms; both memory-bandwidth bound, identical
`f64x2.add`), `cmp` is near-tie (0.23 vs 0.214 ms), and `sum` favors AS by ~18 %
(0.27 vs 0.33 ms) — but AS's own *scalar* `sum` beats both, i.e. the delta is
implementation tuning, not a language ceiling. Both compile to the *same* wasm SIMD
ISA; neither has a throughput ceiling the other cannot reach. Rust's scalar `cmp`
looked terrible (3.13 ms) purely because the spike wrote a per-element memory
read-modify-write; a register-accumulating version (as its own SIMD scalar-tail already
does) closes the gap. **Verdict: throughput does not separate the candidates.**

**Criterion 2 — Binary size: AS wins, but non-binding.** AS is smaller (500 B/582 B vs
673 B/910 B), yet **both are ~84×–132× under the 75 KB per-binary budget.** Even
projecting large growth for the full kernel set plus hash tables and sort, neither is
at risk of the size gate. Size cannot decide this. (Constraint carried forward: Rust
kernels **must** stay `no_std` + `panic="abort"` to keep this headroom — see below.)

**Criterion 3 — Maintainability for agents writing many kernels: decisive, → Rust.**
With criteria 1 and 2 not separating the candidates, the tiebreaker decides, and Rust
wins clearly for a *many-kernel, agent-authored* library:

- **Single-source SIMD gating.** `#[cfg(target_feature="simd128")]` keeps one file per
  kernel covering both builds. AS has no preprocessor and **requires duplicated source
  files** (the spike literally maintains `scalar.ts` + `simd.ts`); across dozens of
  kernels × 2 builds that doubles the surface area and invites scalar/SIMD drift.
- **Standard library for the relational kernels.** Phase-2 Agent C (argsort, top-k) and
  Agent D (64-bit hashing, hash groupby, hash join — ADR-005) need hash tables and
  sorting. Rust provides tested algorithms via `core`/`alloc`/crates; AS has **no**
  standard library, so every hash map and sort is hand-rolled. This is the single
  largest correctness/maintenance risk, and it favors Rust decisively.
- **Native testability & smaller error surface.** Rust kernel logic is unit-testable
  natively before wasm; its explicit integer widths and larger LLM training corpus
  reduce the rate of subtle bugs for AI authors. (The spike's SIMD null bug shows these
  kernels are subtle in *either* language — reinforcing that Rust's testing story is an
  asset, and that null conformance fixtures are mandatory regardless.)

This matches ADR-007's planning expectation ("Rust wins codegen [control]; AS wins
size") and follows the locked order: throughput ties at the ceiling, size is
non-binding, maintainability breaks the tie for **Rust**.

## Consequences & constraints carried forward

- `/wasm/` uses a Rust crate (`crate-type=["cdylib"]`, `wasm32-unknown-unknown`).
  Dual builds per ADR-004: scalar (default) and SIMD (`RUSTFLAGS="-C
  target-feature=+simd128"`), each post-processed with `wasm-opt -O3`
  (`--enable-simd` on SIMD). One source file per kernel family, SIMD gated by
  `#[cfg(target_feature="simd128")]`.
- **Size discipline is a hard requirement, not a default:** keep `no_std`,
  `panic = "abort"`, `lto = true`, `codegen-units = 1`, `opt-level = 3`
  (drop to `"s"`/`"z"` only if a future binary approaches budget). Do **not** pull in
  `std`, formatting, or panic-unwinding machinery — that is what would blow the 75 KB
  budget. CI size gate (§5) guards this on every merge.
- **Null-path SIMD is a known trap.** Reductions/elementwise kernels that special-case
  "all-valid" fast paths must have their null fallback covered by conformance fixtures
  (null-in-pair, all-null, boundary bytes). The corrected spike pattern: never
  `splat` a single value into a 2-lane accumulator; accumulate singles scalar-side.
- No wasm-bindgen: exports are `#[no_mangle] extern "C"` flat functions per ADR-008;
  JS holds typed-array views over `memory.buffer` (ADR-001). The allocator is a spike
  bump allocator; Phase 1 replaces it with a real arena (alloc/free/realloc).

## Reversal clause

Reversing this decision after merge requires a new ADR with benchmark evidence that
Rust fails a §5 gate that AssemblyScript would pass. The spike artifacts
(`wasm/spike/{rust,as}/`), fresh `results.json`, and `verify.mjs` remain in-tree as the
evidentiary baseline.
